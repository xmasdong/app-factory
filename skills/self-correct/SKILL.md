# /self-correct — 交付前自纠正 pass

> **为什么存在:** 从 16 个历史 app/游戏 build 挖出 111 条「用户不得不纠正 AI」,85% 同一个根:**AI 把抽象 spec 直译成最省事实现,再用最弱证据(编译过/单张截图/模拟器)自证完成**。这个 skill 让 AI **在交给用户前先当自己的 QA**——用固定的问题,推当前产品的答案。

**作用:** 拿 `app/rules/build-constraints.md` 的 **8 条通用约束**(元则 + 8 透镜 + 8 自问句)对**当前产物**逐条自省。这不是过一张写死的检查表——**问题固定,答案你按当前产品现场推理**(释放思考空间)。

**何时触发:**
- `/build` 收尾、`/qa` 开始前、或任何你要说「完成/交付」之前
- 无人值守链里:build→qa 之间**自动跑一遍**,把能自纠的自纠掉,再进 qa
- 用户说「这不对/又错了/我不是说过」时——先回来跑这个,而不是打补丁

**输入:** `app/rules/build-constraints.md`(约束源)+ 当前产物(代码/UI/spec/素材)。
**输出:** `.claude/state/self-correct.json` + 自省报告(每条:发现→自纠了什么→剩什么需真机/用户)。

---

## 怎么跑(逐条自问,诚实作答)

先读 `app/rules/build-constraints.md`。然后对**当前这个产品**,逐条回答 8 个自问句。**关键纪律:答案要针对当前产物具体化,不许答"没问题"敷衍——每条至少给一个"最可能出问题的点"再判它成不成立。**

| # | 自问(来自 build-constraints) | 你要产出 |
|---|---|---|
| 1 | 我的证据撑到哪层(能跑/能用/好用/正确)?差的那层现在补 | 当前每个"完成"主张 → 标它的证据等级;弱的列出来补或诚实降级措辞 |
| 2 | 抽象词我是实现了"它"还是"脑补省事版"?对齐过没 | 找出 spec 里被直译的动词/机制/格式 → 展开成显式序列/判据,不一致就重做 |
| 3 | 我以用户身份从头走一遍,哪步卡住/困惑/想退 | 真走一遍核心路径(能自动就自动/否则真机或要用户验)→ 列卡点 |
| 4 | 每个元素/页面/功能挣得了位置吗 | 扫控件/跳转/首屏坑位 → 挣不到的砍或降级 |
| 5 | 一句话这是什么、多大?工作量对得上吗 | 写下 ONE-LINE + 规模档 → 超范围的划出去 |
| 6 | 有没有"错了就翻车"的硬正确性?依据?谁独立验过 | 列数值/规格/可解性/合规点 → 每个标来源 + 跑独立复核(不带原假设) |
| 7 | 这信息是给用户的价值还是我的内部账 | 扫界面/截图/商店素材/文件 → 内部账(数量/凭证/中间文件/模拟器痕迹)藏起来 |
| 8 | 我在独立评估还是顺着用户/顺着省事滑 | 有没有该说的不同意见/做不到的部分被我咽下去了 |

**自纠 vs 上报**:能自己修的(第 2/4/5/7 大多)当场修;需真机/真环境/用户拍板的(第 3/6 常见)诚实列进"待人验",别假装过了。

## 输出 `.claude/state/self-correct.json`

```json
{
  "checked_at": "<ISO8601>",
  "product_one_line": "一句话这是什么",
  "findings": [
    { "constraint": 1, "issue": "深扒页说'完成'但只有模拟器截图为证",
      "action": "self-fixed | needs-device | needs-user", "note": "已补真机 playthrough / 或 降级措辞为'待真机验'" }
  ],
  "self_fixed_count": 0,
  "open_items": [ "需真机验证的点", "需用户拍板的点" ],
  "verdict": "clean | has-open-items"
}
```

`verdict=has-open-items` 不阻塞自动链(这不是硬门),但**必须**把 `open_items` 带进 `/qa` 的 not_verified 和给用户的收尾里——诚实暴露,别吞。

---

## 规则(守住"释放思考空间")

- **问题固定,答案现推**:8 个自问是稳定的;每个产品的答案不同,别背标准答案、别套上一个项目的结论。
- **不是新硬门**:self-correct 是 pre-empt(自纠),默认不阻塞自动链;唯一会硬挡的是 build-constraints §6 的客观正确性(可解性/数值),那走各自的 sg_app_* 门。
- **诚实优先于好看**:证据不够就说不够(约束1),做不到就说做不到(约束8)。self-correct 的价值全在诚实——自欺=白跑。
- **不写死案例**:发现要针对当前产物具体化,不许 copy 历史项目的具体 bug 当模板。
