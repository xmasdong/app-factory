## ECONOMICS

> **先答「钱从哪来」,再决定这节填多少**(下面的表都是订阅+配额型产品的模板,不是所有产品都适用):
> - **有按次后端成本?**(AI 推理/云函数/第三方 API)→ 填单位成本表;**纯本地/无后端** → 写「无按次成本」跳过。
> - **怎么变现?** 订阅/配额 → 填价格阶梯+反薅;买断 → 只写一次性定价;免费/纯 ASO 引流 → 写「当前不变现,靠 X 活」跳过订阅表;广告 → 填 eCPM 估算。
> - **靠买量还是自然搜索?** 买量 → 要 CAC≤LTV;ASO 驱动 → 只需 LTV 为正,CAC 节标 N/A。
> - **不适用的子表显式写「本产品无 X,因为 Y」**,不硬填、不编数。下面的数字(60% 毛利/4-8 月周期/区域比例)是**默认参考不是法律**,按品类实证调。
>
> ViraSnap 教训(有订阅+配额时才成立): $0.04/张 × 月配额 vs 订阅价 没事先算 → 用户狂刷 = 单 ARPU 倒贴。
> **护栏(有多档订阅时不可协商)**: 月价 / 年价折月 / 月配额 必须单调递增,否则白送配额 = 业务自杀。

### 反薅漏洞清单 (堵真实薅点,不是凑数量)

> **只堵本产品真实存在的「发放价值 vs 收费边界」漏洞**:无补贴/无试用/无免费配额/无广告解锁的产品(如纯买断)→ 显式写「无可套利资源」跳过本节。有多少个真实薅点就列多少个,每个配具体防护——**不为凑 5 条编造不存在的漏洞**。
> ViraSnap 教训: 7 天免费试用没设防 = 用户开通 → 用满 → 退款 → 重开. 涉及试用/配额/广告解锁时每条必须有具体防护策略.

1. **免费试用退款漏洞** — 用户试用 7 天用满配额后申请退款. 防护: <TBD: 试用期内配额上限 = 免费档配额 + 设备 ID 绑定首次试用记录, 退款过的设备不再发放试用>
2. **多设备共享同账号** — 一个订阅, 全家共享拉低 ARPU. 防护: <TBD: 同时在线设备数 ≤ N (App Store family sharing 例外), 后端 device_id session 强校验>
3. **看广告解锁重复触发** — 广告 callback 被前端篡改双花. 防护: <TBD: SSV (server-side verification), 广告平台签名校验 + 单次 token 防重放>
4. **多账号刷免费配额** — 注册 N 个账号每个领免费额. 防护: <TBD: 设备指纹 (IDFA/AAID + iCloud KeyValue) 反指认 + 手机号验证一次>
5. **越狱/Root 用户伪造 IAP** — 本地 receipt 伪造. 防护: <TBD: 服务端 Apple verifyReceipt / Google Play Developer API 校验, 永不信任客户端 receipt>
6. **后端配额校验滞后** — 客户端并发请求 N 次, 后端来不及扣配额. 防护: <TBD: 配额扣减用原子操作 (Redis INCR / DB SELECT FOR UPDATE), 客户端预占 + 后端结算>
