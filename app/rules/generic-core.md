# 核心规则

> 本文件定义门禁的完整检查项。触发条件、规模分级和用户覆盖规则见 CLAUDE.md。当两者冲突时，以 CLAUDE.md 为准。

## 四道门禁

每道门禁是硬锁。用 CHECK → IF NOT → IF YES 自检。

### GATE 0: 全局认知

```
CHECK: 能回答以下全部？
- 产品做出来是什么（页面/模块/入口/用户流程）
- 最少需要什么功能，缺什么就不成立
- 用户是谁，使用场景（有完整的用户故事：谁、在什么情况下、怎么用、期望什么结果）
- 商业模式（免费/付费/待定）
- 技术边界（能做/不能做/需要外部依赖）
- 明确不做什么（边界清单）
IF NOT → 带 2-3 个备选方案问用户，不猜
IF YES → 可以拆任务
IF 接手已有项目/外部文档 → 所有声称的完成度视为"可查"，不视为"已知"。必须验真 + 输入质量分级后才能进入拆任务（详见 /spec skill "接手已有项目"）。
OUTPUT → docs/spec.md
```

**GATE 0 是主动的，不是被动的。** AI 不能只等用户提供信息，必须：
1. 主动问用户要参考产品/截图/灵感来源——一张图胜过十段描述
2. 主动生成"不做清单"让用户确认——范围不是用户划的，是 AI 提议、用户确认的
3. 主动扫描技术依赖，标出高风险项（平台 API、第三方服务、特定系统版本），要求每个都有兜底方案
4. 主动把模糊描述转成具体方案让用户选——用户说"做个 App"，AI 要追问到"几个页面、什么导航、什么数据"
5. 基于对这类项目的领域知识，提出用户可能没想到的关键维度——带默认方案提，用户只需否决不对的
6. **涉及 UI 时，主动采集视觉方向**（见下方视觉规格检查项）
7. **主动补充 PRD 没覆盖的内容到 spec 中**——PRD 挑战和故障想象力发现的缺口，AI 直接补充完整定义、状态流转、验收标准和 TASK，不问"要不要补"。用户在审 spec 时否决不需要的部分即可。

**GATE 0 的默认姿态是"补充后审"，不是"确认后补"。** AI 基于领域知识和结构化分析产出的补充内容，大多数情况下比人临时想的更全。等人逐条确认再补是低效的——人的精力应该花在审"AI 补的对不对"，而不是逐条回答"要不要补"。只有用户明确否决时（"这个不需要"、"这个 deferred"）才从 spec 中移除。

#### 前置人工动作清单（必填，全项目适用）

**目标：把用户必须亲自做的事情一次性前置到 GATE 0，后续执行不再中断等人做本机动作。**

典型症状（发生了说明 GATE 0 没做全）：
- AI 做到第 N 个任务才发现"用户得先装 ffmpeg"
- AI 实现一半才意识到"需要 HuggingFace Token"
- 发布前才发现"需要申请 staging 服务器"

这些是 spec 漏项，不是"到那一步才知道"——在 GATE 0 扫描时 AI 完全可以基于技术栈+外部依赖预判出来。

**GATE 0 必须主动扫描并聚合以下 6 类前置动作：**

| 类别 | 例子 | 必填字段 |
|------|------|---------|
| 环境工具 | `brew install ffmpeg`, node 版本, python 版本 | 安装命令 + 验证命令 |
| 凭证/Token | HuggingFace/OpenAI/Anthropic/GCP Key | 申请入口 URL + 填到哪（env 变量名） |
| 账号/服务 | 云账号开通, 域名注册, 应用商店开发者账号 | 申请入口 + 预估等待时间 |
| 权限/审批 | 法务评审, 合规审批, 预算批准 | 对接人 + 预估等待时间 |
| 本机预演 | bootstrap/preflight 脚本在本机跑通 | 命令 + 成功标志 |
| 数据准备 | 训练数据/测试音频/样本文件 | 来源 + 存放路径 |

**输出到 spec.md 顶部（TASK 清单之前）：**

```markdown
## 前置人工动作清单

> 在 AI 开始 /impl 前，这些必须由用户亲自完成一次。完成后说"前置已就绪"即可。

### A. 环境工具（本机装好）
- [ ] `brew install ffmpeg`（验证：`ffmpeg -version`）
- [ ] `brew install uv`（验证：`uv --version`）

### B. 凭证/Token（AI 无法代办）
- [ ] HuggingFace Token（申请：https://huggingface.co/settings/tokens，写入 `.env` 的 `HF_TOKEN=`）

### C. 本机预演
- [ ] `./scripts/bootstrap.sh --preflight` 通过

### D. 资源/账号/审批（如有）
- [ ] 无

**与具体任务的映射：**
- 任务 T3（[CRITICAL] A/B 双跑）→ 依赖 A, B, C
- 任务 T8（数据层）→ 依赖 A
```

**硬规则：**
1. **每条前置动作必须映射到至少一个任务**（说明为什么需要）；没有任务依赖的动作 = 多余，删掉
2. **每个任务的 HUMAN 字段增加 `action:<前置清单项>` 标注** — 表示该任务依赖某前置动作，未完成则阻塞
3. **AI 在 /impl 执行过程发现新前置动作** = spec 没做全 → 回 GATE 0 补充清单，不在任务里临时暴露
4. **用户执行前置动作后统一说"前置已就绪"** — AI 接着按清单验证每条通过，才开始第一个任务
5. 简单项目（纯文档/单文件改动）可写"无前置动作"，但必须显式声明
6. **反扫时机：TASK 草稿完成后再反扫**（/spec SKILL.md Step 5.5），不在 TASK 之前扫——前置动作的精确来源是 TASK.SMOKE/ACCEPT 的具体动作
7. **反扫白名单严格，不是任何 "install" 都算人动作** — 白名单见 /spec Step 5.5；命中白名单但 AI 可自动化（如 CI runner 预装） → 用逃生舱 `HUMAN: 无(AI 自动化: <理由>)`，理由必填

**主动扫描触发词（AI 看到 spec 含这些就必须扩"前置人工动作清单"）：**
- 外部模型/库需要下载：HuggingFace, OpenAI, Anthropic, Claude, HF_TOKEN
- 音视频：ffmpeg, whisper, audio
- 机器学习：torch, cuda, MPS, pyannote, transformers
- 支付/邮件/短信：Stripe, SES, Twilio
- 云服务：AWS, GCP, Azure, Vercel, Netlify
- 包管理非默认：pnpm, uv, poetry, bun
- 移动/桌面签名：Apple Developer, Google Play Console

#### 视觉规格（UI 项目必填，非 UI 项目填"不适用"）

涉及前端/UI 的任务，GATE 0 必须完成两层视觉定义。缺失时不允许进入 /impl。

**第一层：设计系统（DESIGN.md）— 定义"组件长什么样"**

DESIGN.md 是项目的视觉真相源（single source of truth），遵循 [Google Stitch 格式](https://stitch.withgoogle.com/docs/design-md/format/)的 9 个标准章节，精确到像素级。AI 实现任何 UI 组件时必须参考此文件。

获取方式按优先级路由：

```
用户指定了参考品牌？
  → 有（如"做成 Stripe 风格"）→ npx getdesign@latest add <brand>，拉取到项目根目录
  → 无，但有设计稿/截图？→ AI 基于设计稿提取设计 token，生成项目专属 DESIGN.md
  → 无，也没设计稿？→ AI 基于项目类型推荐 2-3 个参考风格（附品牌名 + 一句话风格描述），
    用户选定后拉取参考 DESIGN.md，再根据项目需要调整（改品牌色、调间距等）
```

DESIGN.md 的 9 个章节：

| # | 章节 | 定义什么 |
|---|------|---------|
| 1 | Visual Theme & Atmosphere | 整体风格调性、设计哲学 |
| 2 | Color Palette & Roles | 每个颜色的 hex 值 + 语义角色（主色/强调色/中性色/状态色） |
| 3 | Typography Rules | 字体族、完整层级表（字号/字重/行高/字距） |
| 4 | Component Stylings | 按钮/卡片/输入框/导航/徽章的完整样式（含各状态） |
| 5 | Layout Principles | 间距系统、网格、留白哲学 |
| 6 | Depth & Elevation | 阴影层级表 |
| 7 | Do's and Don'ts | 视觉护栏（必须做/不能做） |
| 8 | Responsive Behavior | 断点、触控目标、折叠策略 |
| 9 | Agent Prompt Guide | 给 AI 的快速颜色参考 + 组件级 prompt 示例 |

**参考品牌的 DESIGN.md 不能直接用，必须定制。** 拉取后 AI 需要调整：
- 品牌色替换为项目自己的色彩（没有品牌色时 AI 基于行业和用户偏好提议）
- 删除项目不需要的组件（如参考品牌有暗色模式但项目不做）
- 补充参考品牌没有的组件（如参考品牌没有表单但项目需要）

**第二层：页面规格 — 定义"每个页面放什么组件"**

DESIGN.md 定义了设计 token 和通用组件，但不知道"团购详情页"应该展示哪些字段。页面规格在 spec.md 中定义，补充 DESIGN.md 覆盖不了的业务视觉层：

```
每个核心页面必须定义：
□ 组件清单：页面由哪些组件构成（用 DESIGN.md 中的组件名引用）
□ 数据映射：每个组件展示什么业务数据（字段名 + 来源）
□ 状态定义：空状态 / 加载态 / 错误态 / 无权限态各展示什么
□ 交互行为：关键交互的触发条件和结果（点击、滑动、长按）
□ 视觉不做清单：明确不要的风格（动画、渐变、某种配色等）
```

**判断标准：把 DESIGN.md + 页面规格交给另一个 AI，它能不能不问用户就开始写页面？能 → 充分。不能 → 不充分。**

**DESIGN.md 在流程中的位置：**

| 阶段 | DESIGN.md 的作用 |
|------|-----------------|
| GATE 0 /spec | 生成或拉取 DESIGN.md + 定制。spec.md 中写页面规格引用 DESIGN.md 组件 |
| /impl | AI 实现 UI 组件时**必须参考 DESIGN.md**。颜色、间距、字号以 DESIGN.md 为准，不自由发挥 |
| /check | 检查实现的 CSS 值是否与 DESIGN.md 一致（颜色、字号、间距） |
| /verify | 视觉一致性归为"真不可知"（需人肉看），但 DESIGN.md 的 Do's and Don'ts 可作为 checklist |

**如果项目安装了设计技能（如 Pencil MCP），DESIGN.md 的定制和页面规格的视觉细化通过设计技能完成。** 没有设计技能时 AI 直接在 markdown 中工作。

充分的理解：
"物品价值追踪 App，4 个 tab：物品列表、账本、统计、设置。
用户拍照后自动抠图。iOS 用 Vision，Android 用 ML Kit。
没有原生能力的设备 fallback 到 API。先免费，不预留付费。
目标用户是个人，给自己朋友用，不上架。
不做：用户注册、社交分享、应用商店上架。"

不充分的理解：
"做一个记录物品价值的 App"
→ 缺：有哪些页面？4 个 tab 放什么？抠图方案？没能力的设备怎么办？不做什么？

**判断标准：如果把你的理解交给另一个 AI，它能不能不问用户就开始做？能 → 充分。不能 → 不充分。**

#### 数据契约（多仓库 / 前后端分离项目必填，单体项目写"不适用"）

前后端之间的高风险字段必须在 spec.md 的"数据契约"章节声明：

```
| 字段 | 类型 | 单位/格式 | 生产者 | 消费者 |
|------|------|----------|--------|--------|
| price | int64 | 分 | backend /api/item | admin-web ItemList.vue, miniapp item-detail.vue |
| created_at | string | ISO8601 | backend | admin-web, miniapp |
| order_status | string | 枚举: pending/paid/shipped/completed/cancelled | backend | admin-web OrderList.vue, miniapp order-list.vue |
```

**高风险字段定义**（必须声明的字段类型）：
- 金额类 → 声明单位（分/元/cent）
- 时间类 → 声明格式（Unix 秒/Unix 毫秒/ISO8601）
- 状态枚举 → 声明值域（列出所有合法值）
- ID 类 → 声明类型（int64/string/uuid）

**数据契约 FROZEN by default：** 变更走 GATE 0 重新评估，不在 /impl 中默默改。变更时 IMPACT 必须列出所有消费方。

#### 外部文档对账（有外部文档时强制，无则跳过）

当用户提供了外部文档（流程图、PRD、设计稿、第三方 API 文档）时：

```
1. 文档登记：
   SOURCE: [文档名] — [类型：流程图/PRD/截图/API 文档] — [获取方式]

2. 核心业务逻辑逐条提取（辅助逻辑按页面/模块批量引用）：
   EXTRACT: [S-001] [文档名#位置] — [一句话描述]
   编号连续，不允许跳号。

3. 每条映射到 spec.md：
   MAP: [S-001] → spec.md#[位置] | DEFER: [理由] | OUT_OF_SCOPE: [理由]

4. 未提取声明（强制，防遗漏）：
   NOT_EXTRACTED: [文档名#区域] — [理由：纯描述性/非功能性/与本期无关]

5. spec.md 增加"外部文档覆盖"章节（表格形式），覆盖率统计。
```

提取结果和 NOT_EXTRACTED 在第一个检查点由用户确认。未标注状态的条目 = spec 缺口，阻塞进入 /impl。

**脚本自动验收（`skill-gate spec` 检查项）：**
- 有 SOURCE 引用但无"外部文档覆盖"章节 → 阻塞
- 覆盖表中有未标注状态的条目 → 阻塞
- 编号不连续 → 警告

#### PRD 挑战（GATE 0 必填，故障想象力前置）

**为什么需要这一步：** spec 不是翻译 PRD，而是挑战 PRD。PRD 的缺口不在已写内容的排列组合里，而在作者没想到的维度——未定义的状态转换、边界行为、角色间不一致、时序假设、数据归属不清。这五类是需求工程中已验证的高频缺陷来源，对着它们逐一检查就是对 PRD 的深度测试。

**方法：五个结构化视角逐一探查，每个视角产出具体的 PRD 缺口。**

```
对每个核心功能，逐一检查以下视角：

### 1. 状态完整性
列出每个业务实体的所有状态 + 所有合法状态转换。
PRD 未定义的转换 = 缺口。

示例（活动报名）：
- 活动状态：草稿 → 上线 → 满员 → 结束。PRD 未定义：满员后有人取消，回到"上线"还是保持"满员"？
- 报名状态：未报名 → 已报名 → 已取消。PRD 未定义：取消后能否重新报名？

### 2. 边界条件
对每个数值/集合/时间：0 时、1 时、上限时、超限时的行为。
PRD 未定义的边界 = 缺口。

示例：
- 0 人报名时列表页显示什么？PRD 只画了有数据的 UI
- 刚好第 10 人（最后名额）需要特殊提示吗？
- 活动名称 200 字时排版会怎样？

### 3. 多角色一致性
对每个写操作：其他角色实时看到什么？延迟看到什么？
PRD 未定义的跨角色行为 = 缺口。

示例：
- 组织者修改活动时间 → 已报名用户看到旧时间还是新时间？
- 管理员禁用用户 → 该用户正在填写的表单会怎样？

### 4. 时序敏感
对每个多步流程：每一步超时/中断/重试时的行为。
PRD 未定义的中断恢复 = 缺口。

示例：
- 报名接口超时，用户重试 → 会不会产生两条记录？
- 用户填写一半关闭页面，重新打开 → 表单数据还在吗？

### 5. 数据生命周期
对每个数据实体：何时创建/修改/软删/硬删/归档？谁有权操作？
PRD 未定义的生命周期节点 = 缺口。

示例：
- 活动结束后报名记录保留多久？
- 用户注销后，其报名记录如何处理？
```

**每个缺口必须有处置（三选一）：**
- `补入 spec` — **立即补充**，不等人确认。补充内容必须包含：具体定义（状态流转/边界值/处理规则）+ 对应的 ACCEPT 验收标准或新 TASK。
- `deferred` — 当前不处理，给出理由
- `不适用` — 该视角不涉及本功能，给出理由

**`补入 spec` 的自动补充原则：**

AI 发现缺口后直接补到 spec 中，不问用户"要不要补"。原因：AI 通过结构化视角发现的缺口往往是人没想到的维度，等人确认只是多一次往返——人在最终审 spec 时自然会看到补充内容，不认可时删除或改为 deferred 即可。

补充内容的质量标准：
- **状态缺口** → 补完整的状态转换定义（从什么状态 → 什么状态 → 触发条件）
- **边界缺口** → 补具体的边界值和处理规则（上限多少、超限怎么处理、前端如何校验）
- **并发/时序缺口** → 补防护机制（锁策略、幂等、宽限期）+ ACCEPT 中写可测试的并发断言
- **每个补充必须落地为 TASK 或追加到已有 TASK 的 ACCEPT 中** — 只写文字定义没有 TASK 承接 = 没补

**只有用户明确说"不需要"时才改为 deferred。** 默认姿态是补入，不是等人拍板。

**脚本自动验收（`skill-gate spec` 检查项）：**
1. PRD 挑战章节存在
2. 五个视角中至少覆盖 3 个（状态完整性/边界条件/多角色一致性/时序敏感/数据生命周期）
3. 有编号的具体缺口条目
4. 缺口有处置标注（补入/deferred/不适用）
5. 每个视角标题下有 ≥1 个具体缺口（不能空挂标题）
6. "补入 spec"的缺口在全文有 TASK/ACCEPT 承接（机械验证落地）

#### 故障想象力（GATE 0 必填，维度驱动）

**为什么需要这一步：** PRD 挑战从结构视角找缺口——"PRD 没定义什么"。故障想象力从用户视角找风险——"这玩意炸了用户会看到什么"。两者互补：PRD 挑战发现遗漏的规则，故障想象力发现遗漏的防御。

**PRD 挑战的产出直接喂给故障想象力：** 状态转换表里每个转换都可能失败，边界条件每个边界都可能越界，多角色一致性每个操作都可能冲突——这些就是故障的具体锚点，比凭空想故障质量高得多。

**方法：维度枚举驱动，不设固定条数阈值。**

```
步骤：
1. 从 PRD 挑战的产出中提取故障锚点：
   - 每个状态转换 → "这个转换失败了会怎样？"
   - 每个边界条件 → "越界时用户看到什么？"
   - 每个跨角色操作 → "不一致时谁看到错误？"
   - 每个时序节点 → "中断后重试会怎样？"

2. 补充维度枚举（覆盖 PRD 挑战未触及的技术故障）：
   - 故障主体：用户操作、网络、数据库、第三方服务、并发、缓存、权限
   - 故障时机：操作前、操作中、操作后、跨操作
   - 故障表现：数据丢失、数据错乱、静默失败、状态卡死、重复执行

3. 格式：如果这个功能在生产上炸了，新闻标题会怎么写？

要求：
- 写"用户视角的具体糟糕结局"，不是"代码层面的异常"（❌"空指针异常" ✅"未登录用户看到所有活动都显示已报名"）
- 每条必须有主语（哪种用户）+ 具体动作 + 看到的具体画面
- 不允许"同上"、"参考 X"、"通用错误处理"这种抽象逃生路径

示例（活动报名功能）：
1. "未登录用户看到所有活动都显示已报名"（权限 × 操作前 × 数据错乱）
2. "用户报名后刷新页面，状态消失"（缓存 × 操作后 × 数据丢失）
3. "活动满员后第 N+1 个用户仍然报名成功"（并发 × 操作中 × 数据错乱）
4. "用户误报名了一个已结束的活动且无法取消"（状态转换缺口 → 故障）
5. "同一用户重复点击产生 N 条报名记录"（时序 × 重试 × 重复执行）
6. "弱网下提交报名，用户看到成功但后端没收到"（网络 × 操作中 × 静默失败）
7. "用户 A 取消报名后，候补用户 B 没有被自动递补"（跨角色 × 数据不一致）
```

**写完后的对账（强制，不对账视为未完成）：**
- 每条糟糕结局，对照 spec 检查：是否被某个验收标准防住？
- 被防住 → 在该验收标准旁标注 `防 故障#N`（每个编号独立写，不可简写为 `#2 #3`）
- 没防住 → **立即补验收标准到对应 TASK 的 ACCEPT 中**，或追加新 TASK。标注 `防 故障#N`。只有用户明确说"这个不用防"时才写 `deferred`。
- **与 PRD 挑战同理：默认姿态是自动补防御，不是等人确认。** 故障想象力发现的风险如果不落地为验收标准，就只是一份文档——spec 的价值在于每个风险都有对应的 ACCEPT 可测试。

**与覆盖契约的关系：** 故障想象力在 GATE 0 产出，/spec Step 1.6 的"核心链路覆盖契约"消费它——清单里的每条故障新闻标题，自然对应到 stabilizing 阶段的一个 E2E 测试用例。这是 spec → stabilizing 的强制投影。

**项目类型适配：**
- UI 项目：必填，聚焦用户看到的画面
- 纯后端 API：必填，聚焦调用方看到的错误响应 + 数据状态
- CLI/库：必填，聚焦调用者收到的错误结果 + 副作用

**这一步不可跳过。** 没有这一步的 spec 是 happy path 驱动的 spec，下游一切检查都无法弥补最初的盲区。

**脚本自动验收（`skill-gate spec` 检查项）：**
以下由 `stop-skill-gate.sh` Hook 在 AI 宣称 /spec 完成时自动执行，不依赖 AI 自律：
1. PRD 挑战章节存在 + ≥3 个视角 + 有编号缺口 + 缺口有处置标注
2. 故障想象力章节存在 + 维度枚举存在（≥2 个维度头）+ 有编号条目
3. 主语覆盖率 ≥ 60%（正则匹配"用户/管理员/调用者"等主语词，低于阈值打回）
4. 条目去重（去掉末尾差异后 unique 数 < 总数一半 → 判定为套话）
5. 对账标注存在（spec 中必须有"防 故障#N"标注，0 个对账标注 → 打回）
6. 覆盖缺口检测：每条故障必须被至少一个 ACCEPT 标注覆盖

脚本能挡住结构性偷懒（无视角、无主语、套话、跳过对账），挡不住语义性偷懒（有主语但缺口不真实）。后者仍需人在检查点审核。

#### 核心难点识别（GATE 0 拆任务前必做）

**为什么需要这一步：** 任务清单是平铺的，但模块的复杂度不是。一个项目的成败往往取决于 1-3 个核心难点模块——它们是最耗时间、最需要调试、选错方案返工成本最高的部分。如果不在规划阶段就把它们拎出来深化，AI 会把难点模块和普通 CRUD 用同样的深度处理，结果是核心模块带着未经论证的方案直接进入实现，在调试阶段才暴露方案缺陷。

**识别标准（命中任一条 → 标记为核心难点 `[CRITICAL]`）：**

```
□ 多模块协作：≥3 个模块协作完成一个功能（如"下单"涉及库存、支付、订单、通知）
□ 并发/一致性：有竞态条件、分布式事务、幂等性、锁竞争等要求
□ 方案不唯一：实现路径有 ≥2 个截然不同的方案，且各有显著 trade-off
□ 高扇出依赖：其他 ≥3 个模块依赖此模块的输出（此模块的接口变更会级联）
□ 调试密集：需要特定条件（并发、时序、数据量）才能复现正确/错误行为
□ 领域复杂：业务规则本身就复杂（多条件定价、审批流、权限矩阵、状态机 >5 个状态）
```

**核心难点模块必须在 spec.md 中有独立的方案简述：**

```
## 核心难点

### [CRITICAL] 模块名：一句话描述

**为什么是难点：** 命中了哪条识别标准

**方案选择：**
| 方案 | 描述 | 优势 | 劣势/风险 |
|------|------|------|----------|
| A（推荐） | ... | ... | ... |
| B | ... | ... | ... |

**选择理由：** 为什么选 A 不选 B（一句话）

**验证策略：** 怎么在实现早期验证方案可行（不是等全做完才发现不行）
- 最小验证：用什么最小实验确认核心假设？（如：写一个 benchmark 验证锁竞争在目标 QPS 下是否可接受）
- 失败信号：什么现象说明方案走不通？（如：P99 延迟 >500ms，或死锁频率 >1%）
- 回退方案：验证失败时切换到方案 B 的成本？
```

**对任务拆分的影响：**

- `[CRITICAL]` 模块的任务必须**优先排序**——排在任务清单前段，优先实现
- `[CRITICAL]` 模块的第一个任务应该是**方案验证**（最小实验），不是直接写完整实现
- 方案验证通过后再展开后续任务；验证失败则切换备选方案，不在错误方向上堆代码
- `[CRITICAL]` 任务的 TASK-TEMPLATE 中 HUMAN 字段默认为"阻塞"——方案需要人确认后才能开始实现

**没有核心难点的项目：** 不是所有项目都有。纯 CRUD、静态页面、配置调整等项目，写"无核心难点"即可。但这个判断本身必须是显式的——AI 不能跳过这一步。

**脚本自动验收：**
- spec.md 中存在"核心难点"章节（或显式声明"无核心难点"）
- 有 `[CRITICAL]` 标记的模块必须有方案选择表格（≥2 行）
- 有 `[CRITICAL]` 标记的任务在 status.md 的任务清单中排序靠前

### GATE 1: 理解门禁

```
CHECK: 当前任务能回答？
- 一句话说清做什么
- 验收标准可测试
- 预计改哪些文件
IF NOT → 停，问用户，不写代码
IF YES → 进入实现
```

**IMPACT 自检（拆任务时强制）：**
- IMPACT 中列出的消费方，是否已纳入本任务或后续任务的 FILES？未纳入 → 追加任务或说明为什么不需要改
- 改了 API 响应字段 → IMPACT 列所有调用该 API 的前端页面/组件
- 改了 DB 字段/类型 → IMPACT 列所有读写该字段的代码路径
- 改了金额单位/日期格式/枚举值 → IMPACT 列所有格式化/解析该值的代码
- 多仓库项目：IMPACT 必须跨仓库列出

**SMOKE 自检（拆任务时强制，小任务豁免）：**
- SMOKE 必须包含具体的调用方式（curl/浏览器操作/CLI 命令/测试函数）
- 预期结果必须包含业务数据（不是"200 OK"，是"返回包含 price>0 的 JSON"）
- 依赖外部服务 → SMOKE 声明用真实服务还是 mock，mock 时如何确保切换到真实时不静默降级

**实现中范围检查（commit 前强制自检）：**
- 当前改动的文件是否超出 TASK-TEMPLATE 的 FILES 列表？超出 → 停，评估是任务定义不准（更新 FILES）还是 scope creep（拆新任务）
- 正在写的代码是否直接服务于 ACCEPT 中的验收标准？删掉后验收仍通过 → 不写，拆为独立任务
- 实现中遇到 spec 未定义的决策（空状态处理、边界行为等）→ 记入 status.md 为 optimistic/deferred，不默默推断

**Mock/Stub 规则（生产代码路径）：**
- 生产代码中的 mock/stub 必须满足以下至少一项：
  a) 函数/文件名包含 Mock/Stub/Placeholder/Fake 标识
  b) 有环境变量控制开关，非生产环境时输出 WARNING 日志
  c) 在 DONE-TEMPLATE 的 STUB_REMAINING 中显式声明
- 静默降级（返回假数据且不报错不记日志）在生产路径中禁止
- `stub-scan` 脚本在 commit 前和 /impl 验收时自动检测违规

### GATE 2: 能力门禁

GATE 2 有两个时机：**规划时预扫描**（/spec 阶段）和**实现时补漏**（/impl 阶段）。两者用同一套判断逻辑。

#### 测试分层

GATE 2 的"能写测试吗？"包含三个层次，按项目阶段逐步启用：

| 测试层 | 验证什么 | 启用时机 | AI 职责 |
|--------|---------|---------|---------|
| 单元测试 | 函数/模块正确性 | 始终 | 写测试 + 跑测试 |
| 集成测试 | 模块间交互、API 契约 | 始终 | 写测试 + 跑测试 |
| E2E 测试 | 核心业务链路端到端跑通 | stabilizing 起 | 写脚本 + 跑脚本，CI/环境配置生成模板由人审核 apply |

**building 阶段不要求 E2E。** 原因：核心流程还没跑通时，E2E 维护成本高于收益，随架构变动频繁失效。building → stabilizing 的切换条件本身（"核心用户流程首次端到端跑通"）就是 E2E 的第一个用例。

**E2E 的前置条件：**
1. spec.md 中已定义"覆盖契约"（核心链路 + 显式不覆盖链路，/spec Step 1.6 产出）
2. 有可运行的测试环境（本地模拟 或 staging）
3. 前置条件不满足时标记为 `deferred`，不阻塞其他任务

**覆盖契约（FROZEN by default）：** 业务链路组合无界，"完整覆盖"是伪命题。/verify 不回答"还有没有遗漏"——它只对照 spec.md 的覆盖契约判断是否完成。新增链路 = 修改契约 = 回 /spec，不在 /verify 内吸收。详见 /spec Step 1.6 与 /verify。

**E2E / 冒烟测试形式选择（AI 自主决定，不问人）：**

根据项目技术栈和测试环境，AI 按以下路由选择测试形式：

| 项目类型 | 默认测试形式 | 工具 | 存放位置 |
|---------|------------|------|---------|
| 纯后端 API | HTTP 级别脚本，直接调接口走完链路 | 项目自带测试框架（Go: `testing`, Node: `jest`/`vitest`） | `tests/e2e/` |
| 前端 + 后端 | API 级别优先；涉及前端交互的链路用浏览器自动化 | Playwright / Cypress | `tests/e2e/` |
| 纯前端 / 小程序 | 组件级集成测试 + 关键路径手动测试清单 | 项目自带测试框架 + `docs/smoke-test-checklist.md` | 测试文件同项目，清单在 docs |
| 多端（后端 + 管理后台 + 小程序） | 后端 API 级别脚本覆盖核心链路，前端出手动测试清单 | 同上组合 | 同上 |

**选择原则：** 能用代码自动化的不写手动清单；能在 API 层验证的不拉浏览器；手动清单只用于 AI 无法自动化的部分（真机、小程序容器）。

**AI 的能力边界（硬规则）：**
- AI 可以：写 E2E 测试脚本、生成 CI/CD 配置模板、在本地跑测试命令、检查配置语法
- AI 不可以：推送触发 CI、配置 secrets/凭证、访问云控制台、验证 staging 可达性
- CI/CD 配置和 secret 管理由人完成，AI 生成的配置必须经人审核后手动 apply

#### 第一关：能写测试吗？

```
CHECK: 这个验证项能写测试吗？（参考上方测试分层，选择合适的层次）
IF YES → 写测试，不找人。到此为止。
IF NO  → 进入第二关（回滚成本路由）
```

**把可测试的东西推给人确认 = 偷懒，不是严谨。**

| 假不可知（能写测试，不该找人） | 真不可知（无法用 assert 表达） |
|-------------------------------|-------------------------------|
| 数值正确性（金额、数量、百分比） | 视觉审美（配色、布局、间距是否好看） |
| 数据格式（日期、货币符号、单位） | 真机体验（权限弹窗、系统兼容、设备差异） |
| 状态流转（按钮禁用/启用、页面跳转） | 业务意图（spec 中未定义的规则） |
| API 返回值（字段存在、类型正确） | 交互感受（动画流畅度、操作直觉性） |

**注意：业务意图只有在 spec 中未定义时才是"真不可知"。** 如果 spec 明确写了规则（如"满 3 次打折"），那就是可测试的——写断言验证实现是否符合 spec。只有 spec 没覆盖到的业务问题才需要问人。

**测试前置条件：** "能写测试"要求测试的期望值来自已确认的来源（spec、已确认的数据契约、已有代码行为）。如果期望值本身是假设（如"后端返回的是元还是分"），那不是"能测试"——那是"期望值本身是一个待确认的决策"。先确认契约，再写测试。

#### 第二关：回滚成本路由

确认是"真不可知"后，用**结构信号**判断回滚成本——不靠主观感觉，靠可检查的事实：

```
结构信号检查（任一命中 → 高回滚成本）：
□ 决策影响 >1 个文件
□ 涉及数据模型、API 契约、或跨组件接口
□ 已有代码依赖此决策
□ 涉及不可逆操作（数据库迁移、外部 API 调用、支付）

全部未命中 → 低回滚成本（纯展示层：CSS、文案、配色、间距）
```

| 回滚成本 | 是否阻塞后续任务 | 处理方式 | 示例 |
|---------|----------------|---------|------|
| 低 | — | **乐观执行**：先按最合理的方案做，记入 status.md 待确认清单，在检查点批量让人确认 | 配色、间距、文案语气 |
| 高 | 否 | **跳过先做别的**：标记为 `deferred`，优先做不依赖此决策的任务 | 数据模型选择、第三方服务选型 |
| 高 | 是 | **停下来等人**：这是唯一需要阻塞的情况 | 核心业务规则、支付流程、不可逆操作 |

#### 决策生命周期

**状态定义**（每个人确认项有且仅有一个状态）：

```
optimistic   — 已按默认方案实现，等待检查点确认
confirmed    — 用户已确认，AI 可依赖此决策执行
deferred     — 未确认，已跳过，不阻塞其他任务
invalidated  — 原决策被推翻，依赖此决策的代码需要回滚或重做
fused        — 熔断跳过，AI 连续修复失败后放弃的任务（等待人工介入）
```

**状态流转：**
```
乐观执行 → optimistic → 检查点确认 → confirmed
                                   → invalidated → 回滚评估
跳过     → deferred   → 用户决定后 → confirmed（解锁被阻塞的任务）
                                   → invalidated
阻塞等待 → （无状态）  → 用户决定后 → confirmed（继续执行）
```

**状态归属：spec.md 记录初始状态，status.md 记录状态变更。** 查当前状态看 spec.md，查变更历史看 status.md。

**`invalidated` 回滚评估**（具体规则，不靠判断）：
1. 列出所有依赖此决策的文件（通过 TASK-TEMPLATE 的 DEP 字段 + 代码搜索）
2. 影响 ≤ 3 个文件 → 最小修复 + retest
3. 影响 > 3 个文件或涉及数据模型 → 回到 GATE 0 重新评估
4. 存在链式依赖（决策 A → 决策 B → 任务 C）→ 逐级检查，所有下游决策标记为 `invalidated` 直到确认不受影响

**`deferred` 项老化规则：**
- 每次批量检查点时，重新评估所有 `deferred` 项：是否仍然不阻塞？是否已经被实现代码隐式决定了？
- 如果一个 `deferred` 项被 3 个以上已完成的任务隐式依赖 → 升级为阻塞项，必须立即让用户决定

#### 冻结边界（FROZEN）

**confirmed 的决策可以进一步升级为 FROZEN——不可再讨论的约束。** 冻结把高频决策变成零成本查表。

**什么时候冻结：**
- 多个任务已依赖此决策（改动成本远大于当初决策成本）
- 用户明确要求锁定
- 涉及外部契约（API 已发布、数据模型已有生产数据）

**在 spec.md 中标记：**
```
## 冻结边界

FROZEN: 数据模型 — User 表结构（2026-04-01 confirmed，任务 1-5 依赖）
FROZEN: API 契约 — /api/v1/* 响应格式（已发布，外部消费者依赖）
FROZEN: 技术选型 — 状态管理用 Riverpod（全局依赖，切换成本 > 重写）
```

**冻结的硬规则：**
- FROZEN 项不能被 optimistic 执行覆盖——任何涉及 FROZEN 项的变更必须回到 GATE 0
- 解冻需要用户明确指令 + 影响评估（列出所有依赖此决策的文件和任务）
- `ai-rules.sh scope` 在检测到改动涉及 FROZEN 声明的文件时输出警告

**FROZEN vs confirmed：** confirmed 是"用户说了 OK"，FROZEN 是"改动成本已经高到不值得重新考虑"。大多数 confirmed 不需要冻结——只有被广泛依赖的决策才值得。

#### 批量检查点

不逐项打断用户。**事件驱动触发**：

1. **/spec 完成时** — 所有预扫描出的人确认项，一次性让用户过
2. **依赖触发** — 当下一个任务的 DEP 引用了 `optimistic` 或 `deferred` 项时，该项必须在任务开始前确认
3. **累积阈值** — `optimistic` 项累计 ≥ 5 个时，强制触发检查点，不等自然节点
4. **里程碑节点** — 一个功能模块全部任务完成时

检查点输出格式：
```
## 待确认项（共 N 项）

### 乐观执行（已按默认方案实现，确认或否决即可）
1. [页面/位置] — [具体判断问题] — 否决影响：[涉及的文件列表]

### 待决策（跳过未做，确认后解锁后续任务）
1. [决策问题] — [2-3 个备选方案] — 阻塞的任务：[任务编号列表]

### deferred 项复查（仍然可以继续跳过吗？）
1. [决策问题] — 已被 N 个任务隐式依赖 — 建议：[继续跳过/立即决定]
```

#### 确认项质量要求

**每个确认项必须可执行：**
- **看什么**：具体页面/位置，仅限实际改动过的（没改的不要列）
- **判断什么**：不是"确认正确"，而是具体的判断问题（如"这个间距看起来是否太窄？"）
- **怎么到达**：操作步骤
- **否决影响**：如果用户说不对，需要改哪些文件

**判断标准：用户拿到确认项后，能不能不问 AI 就完成确认？能 → 合格。不能 → 不合格。**

### GATE 3: 状态门禁

```
CHECK: 以下全部完成？
- 代码已 commit（不攒多个任务）
- docs/status.md 已更新
- 关键决策已写入文件（不只在对话里）
- 放弃的方案已记录（试了什么、为什么放弃、换成了什么）
- CLAUDE.md 包含不变的运维知识（部署流程、数据契约、环境信息）
IF NOT → 先完成再开始新任务
IF YES → 可以继续
```

**失败记忆（放弃的方案）是 GATE 3 的强制项。** 原因：AI 最常见的浪费是在不同会话中重复尝试已被证明不可行的方案。status.md 的"放弃的方案"章节是跨会话的负面缓存——记录什么路走不通，比记录什么路能走更能防止浪费。

记录格式（在 status.md `## 放弃的方案` 章节）：
```
- [方案名]：尝试了 [具体做法]，失败原因 [具体原因]，改用 [替代方案]
```

**什么必须记录：** 花了 >10 分钟尝试的方案、用户否决的方案、技术上不可行的方案。
**什么不用记录：** 第一眼就排除的选项、纯代码层面的 bug fix 尝试（这些在 git log 里）。

**大项目警告：项目越大，GATE 3 越关键。** 大项目一定会上下文溢出。溢出时，对话里的信息全部丢失，只有文件能存活。如果 status.md 没有及时更新，每次溢出就是一次进度归零。

status.md 的自检标准：**如果下一秒上下文被清空，新会话只读 CLAUDE.md + status.md，AI 能不能不问用户就继续工作？**

CLAUDE.md 放不变的知识（部署流程、数据契约、环境地址），status.md 放变化的进度（任务状态、下一步、放弃的方案）。两者配合，溢出恢复成本归零。

**脚本自动验收（`skill-gate impl` 检查项）：**
以下由 Hook 在 AI 宣称 /impl 完成时自动执行：
1. `PROJECT_PHASE` 字段存在且值合法（building/stabilizing/live）
2. stabilizing/live 阶段：`## 放弃的方案` 章节必须存在
3. status.md 更新新鲜度：最近 N 个 commit 是否包含 status.md 变更（连续 3 个 commit 未更新 → 打回）
4. 至少有一个已完成的任务（`- [x]`）
5. 最近有 commit（10 分钟内）

#### 结构信号教训捕获（docs/lessons.md）

**不让 AI 自证。** 让 AI 在 GATE 3 时自问"我原本以为 X 实际是 Y 吗"是 LARP——LLM 没有 episodic memory，任务通过的正反馈会抹掉一切"我曾经错过"的叙事。教训不能由犯错的人自报，必须由日志系统观测。

**由 `scripts/ai-rules.sh lessons` 脚本扫描结构信号自动生成，不由 AI 反思写入。** 扫描源全部是可机械检测的外部触发：

| 信号源 | 含义 | 对应"我原本以为 X 实际是 Y" |
|--------|------|--------------------------|
| status.md 中 `invalidated` 状态的决策 | 原决策被推翻 | 最直接的"原本以为正确的决策" |
| status.md "放弃的方案" 章节 | 显式记录的试错 | 哪条路不通 |
| git log: commit 修改了前 N 个 commit 刚加的代码 | 短期返工 | 实现思路不稳定 |
| 任务实际 FILES vs TASK-TEMPLATE 声明偏差 > 50% | 范围估计错误 | 任务边界预判失误 |
| optimistic → invalidated 转换 | 乐观执行翻车 | 默认方案假设不成立 |

**写入格式**（脚本生成，一行一条，append-only）：
```
[日期] [phase] [信号类型] [一句话摘要] [相关文件]
```

**消费端高频化——这是机制成立的关键：**
- lessons.md **不是 phase 切换才读**，那样消费频率太低会导致写入端衰减
- **每次 /spec 开始时强制 grep** — 任何新需求评估前，先查历史教训里有没有相关条目
- /spec SKILL.md Step 1 的第一个动作就是 `grep -i "关键词" docs/lessons.md`

**为什么不放在 GATE 3 让 AI 自填：** 任何让 AI 自证的机制都是脆弱的。/spec 读 lessons.md 是外部约束驱动的强制消费，和 AI 自问不同。

## 项目阶段（PROJECT_PHASE）

方法论是有状态的。`docs/status.md` 必须声明当前阶段，所有技能根据阶段自动调整默认行为。

### 三个阶段

```
building      — 从零构建，核心功能尚未跑通
stabilizing   — 核心功能已跑通，在补测试、修 bug、打磨体验
live          — 已上线/交付，只做维护和迭代
```

### 切换条件（可检查的事实，不靠主观判断）

| 从 → 到 | 触发条件 |
|---------|---------|
| building → stabilizing | 核心用户流程首次端到端跑通（可演示） |
| stabilizing → live | /release 首次输出 `release-ready` 或 `ready-for-staging` |
| live → building | 用户声明开始重大重构（手动切换） |

### 阶段对技能的影响

| 技能 | building | stabilizing | live |
|------|----------|-------------|------|
| /spec | 允许大范围探索 | 限制在已有架构内 | 仅增量变更 |
| /impl | 允许架构变动和重构 | 仅在测试覆盖下改动 | 最小改动，必须有回滚路径 |
| /check | 功能正确性为主 | 加入整体连通性确认 | 加入影响范围评估 |
| /verify | 通过即可 | 加入回归测试检查 + E2E 验证 | 必须评估生产影响 + E2E 全链路 |
| /release | 代码通过即可 | 核心链路必须有可运行验证 | 强制要求 staging 全链路通过 |
| GATE 2 | 乐观执行阈值正常 | 高回滚成本项强制人确认 | 所有非展示层变更需确认 |
| E2E 测试 | 不要求，可选 | 核心链路必须有，**每条 GATE 0 故障想象力清单项必须对应一个 E2E 用例**，CI 集成推荐 | 每次 release 前强制，失败即阻塞 |
| 视觉方向 | GATE 0 采集 + 设计技能细化 | 仅微调，不改整体风格 | 不改 |

### 新项目默认 building，接手项目根据验真结果判断。

### 无人模式（autonomous）

当任务链在无人值守下运行时（`docs/status.md` 声明 `AUTONOMOUS: true`），以下规则覆盖默认行为：

- optimistic 检查点阈值从 5 提高到 10，触发时写入 status.md 而非等待响应，继续执行
- 高回滚成本+阻塞 → 改为 deferred（跳过先做别的），不停下来等人
- deferred 项不因老化规则升级为阻塞——但每项必须记入 status.md 且标注 `[auto-deferred]`
- 全部任务完成或全部被 deferred 阻塞 → 停，输出汇总：已完成列表 + auto-deferred 列表 + 建议的人工决策顺序
- **不可降级的底线**：安全底线 + 不可逆操作（数据库迁移、支付调用、外部 API 写入）仍然硬阻塞，等人处理

## 三类自检

每个决策点，判断当前信息属于哪类：

```
已知（有信息 + 能验证）  → 直接做
可查（读代码/跑测试能搞清） → 先查再做
不可知（AI 能力边界外）   → 停，交给人
```

第三类当第一类处理 = 项目失败最常见原因。

## 质量闭环

```
跑测试 → 失败？→ 分析根因 → 最小修复 → 重跑（最多 3 轮）
         通过？→ 继续
```

修 bug vs 换方案的判断：
- 第 N 轮改动的文件数 > 第 1 轮的 2 倍 → 可能是方案问题
- 修了 A 坏了 B，修了 B 坏了 A → 设计冲突，不是实现 bug
- 需要改的代码超出当前任务范围 → 架构问题

方案问题 → 停，回 GATE 0 重新评估。不在错误方向上反复修。

### 熔断器（Fuse）

质量闭环的"最多 3 轮"由脚本强制执行，不靠 AI 自律。`ai-rules.sh fuse` 在每次 skill-gate 失败时自动累加计数：

```
skill-gate 连续失败 3 次 → 软熔断（跳过当前任务，尝试下一个）
连续 2 个任务触发软熔断 → 硬熔断（停止所有执行，等人）
同一文件被修改 > 5 次   → 软熔断（在两个状态间振荡的强信号）
```

**软熔断响应：**
1. 当前任务在 status.md 中标记 `[FUSED]`
2. 记录熔断原因到 `.claude/state/fuse-state.json`
3. next-task 跳过 fused 任务及其下游依赖
4. 尝试下一个无阻塞、未熔断的任务

**硬熔断响应：**
1. 停止所有执行
2. 输出熔断报告（哪些任务被 fused、为什么）
3. 等待人工介入：`scripts/ai-rules.sh fuse reset`

**重置条件：** commit 成功 → 软重置（gate 失败计数归零）。任务成功完成 → 软熔断计数归零。人工 `fuse reset` → 完全重置。

### Bug 定性前置

**Bug 修复不直接路由到小任务。** 先定位根因层级，再决定规模：

```
收到 bug 报告 →
  1. 定位：是哪层的问题？
     - 展示层（格式化、样式、文案）→ 小任务
     - 逻辑层（计算错误、状态流转）→ 评估影响文件数，≤2 文件小任务，>2 文件标准任务
     - 数据层（数据写错、迁移问题）→ 标准任务，必须有数据修复方案和回滚路径
  2. 修复：按定性结果走对应流程
  3. 回归：评估修改的影响范围，补充对应层的回归测试
```

**live 阶段的 bug 额外规则：**
- GATE 3 的文档更新可在修复 commit 后延迟补完，不阻塞紧急修复
- 其他硬规则全部保留（特别是测试，不能跳过）
- 数据层 bug 必须先评估受影响数据范围，修复代码和数据修复脚本分开 commit

## 交付标准

完成 = 代码 + 测试通过（有实际输出） + 人工确认项状态明确（`confirmed` 或 `deferred` 且已记录在 status.md）。`optimistic` 项必须在下一个检查点确认。`deferred` 项不阻塞任务完成，但必须有记录且不被隐式依赖。

## 安全底线

密钥走环境变量。不 force push。不改冻结契约（spec.md 中标记为 `FROZEN` 的决策）。

## 禁止模式

以下模式现有硬规则未覆盖，命中即违规：

| 禁止 | 识别信号 | 正确做法 |
|------|---------|---------|
| 预建抽象 — 为"将来可能需要"写接口/基类/工厂 | 当前只有一个实现者 | 直接写具体实现，出现第二个用例时再抽象 |
| 幽灵依赖 — 添加 spec/任务未提及的外部包 | package.json/go.mod 等出现任务未要求的新条目 | 停，确认是否需要；能用标准库解决的不引入 |
| 防御性冗余 — 在调用链多层重复做同一校验 | 同一检查逻辑出现在 >1 层且无跨层契约要求 | 选一层负责，文档注明 |
| 范围蠕变 — 实现中"顺手"加功能或重构不相关代码 | 改动与当前 ACCEPT 无关且删掉后验收仍通过 | 记录为新任务，不在当前任务实现 |

## 执行纪律

- 一次一件事
- 卡住就说卡住
- 需求矛盾就问，不猜
- 环境问题 > 3 轮未解决 → 建议开新会话
